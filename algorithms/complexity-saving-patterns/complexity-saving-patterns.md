# 复杂度节省技巧

这篇文档用于沉淀工程和算法中常见的复杂度优化技巧。核心目标不是背结论，而是训练一种判断方式：先观察数据范围、状态空间、访问模式和更新模式，再选择能减少时间或空间成本的表达方式。

## 总体判断框架

做复杂度优化时，可以先问四个问题：

1. 数据范围是否足够小？
2. 状态是否可以被压缩成整数、位、区间或前缀关系？
3. 查询和更新是否有重复模式？
4. 是否需要保持顺序、去重、计数或快速定位？

很多优化都来自这几个问题的答案。下面按常见技巧整理。

## 小范围状态用数组代替哈希表

当 key 的范围比较小、连续或可以映射到较小整数时，优先考虑数组。

典型场景：

- 字符统计：`a-z`、`A-Z`、ASCII 字符。
- 数字出现次数：值域是 `0..100000` 这类明确范围。
- 状态标记：节点编号、用户等级、枚举类型、桶编号。

收益：

- 查询和更新都是稳定的 `O(1)`。
- 常数开销通常小于哈希表。
- 内存布局连续，缓存友好。
- 不需要处理哈希冲突、扩容和对象包装开销。

取舍：

- 值域太大但实际元素很少时，数组会浪费空间。
- key 不连续时，需要额外映射。
- 范围来自外部输入时，要注意边界校验。

示例：

```text
如果只统计小写字母频率：
  用 int[26]

如果统计任意字符串出现次数：
  用 HashMap<String, Integer>
```

## 布尔状态用位图或 bitset 压缩

当只需要记录某个状态是否出现过时，可以用位图代替 `bool[]`、集合或哈希表。

典型场景：

- 判断整数是否出现。
- 标记访问过的节点。
- 表示一个集合状态，例如动态规划中的选取集合。

收益：

- 空间从按字节或对象存储，压缩到按 bit 存储。
- 位运算可以一次处理多个状态。
- 适合做集合并、交、差、包含判断。

常见形式：

```text
visited[x] = true
可以替换为：
mask |= 1 << x

判断是否访问：
mask & (1 << x) != 0
```

取舍：

- 状态数量不能超过整数位数，或者需要使用更大的 bitset。
- 可读性比数组略差，需要清晰命名。
- 对动态增长的大范围 key 不友好。

## 多次区间查询用前缀和

如果需要频繁查询区间和、区间计数或区间某类累计值，可以先预处理前缀和。

典型场景：

- 多次查询数组 `[l, r]` 的和。
- 统计字符串某段中某个字符出现次数。
- 二维矩阵子矩形求和。

收益：

- 预处理 `O(n)`。
- 每次查询从 `O(n)` 降到 `O(1)`。

核心公式：

```text
sum(l, r) = prefix[r + 1] - prefix[l]
```

取舍：

- 适合静态数据。
- 如果中间有频繁更新，需要考虑树状数组或线段树。

## 频繁区间更新用差分数组

如果有很多次区间加减，最后只需要得到最终数组，可以使用差分数组。

典型场景：

- 多次对 `[l, r]` 加一个值。
- 统计多个区间覆盖次数。
- 批量应用范围操作。

收益：

- 每次区间更新从 `O(n)` 降到 `O(1)`。
- 最后一次前缀累加还原结果。

核心操作：

```text
diff[l] += value
diff[r + 1] -= value
```

取舍：

- 适合离线处理。
- 如果更新和查询交替发生，通常需要树状数组或线段树。

## 有序窗口用双指针

当问题涉及连续区间，并且窗口扩大或缩小时某个条件具有单调性，可以考虑双指针。

典型场景：

- 最短或最长连续子数组。
- 滑动窗口内元素满足某个计数约束。
- 排序数组中的两数关系。

收益：

- 暴力枚举区间通常是 `O(n^2)`。
- 双指针可以把左右指针移动总次数控制在 `O(n)`。

判断信号：

- 右指针右移后，窗口某个指标只会单调增加。
- 左指针右移后，窗口某个指标只会单调减少。
- 每个元素最多进窗口一次、出窗口一次。

取舍：

- 不适合没有单调性的任意区间问题。
- 窗口状态维护要非常清楚，例如计数、和、最大值等。

## 重复计算用缓存或记忆化

如果递归、搜索或工程函数中反复计算相同输入，可以加入缓存。

典型场景：

- 递归 Fibonacci、DFS 状态搜索。
- 动态规划的自顶向下写法。
- 工程中重复请求相同配置、元数据或计算结果。

收益：

- 把指数级重复搜索压缩为“状态数 x 单次转移成本”。
- 在工程中可以减少 IO、网络请求或昂贵计算。

取舍：

- 要定义稳定的缓存 key。
- 要考虑缓存生命周期、失效和内存占用。
- 对带副作用的函数不能随便缓存。

## 稀疏数据用哈希，稠密小范围用数组

数组和哈希表不是谁更高级的问题，而是数据分布不同。

选择数组：

- key 是整数或可映射为整数。
- 范围小。
- 访问频繁。
- 需要极低常数开销。

选择哈希表：

- key 很稀疏。
- key 是字符串、对象或组合结构。
- 值域很大但实际元素少。
- 需要动态插入删除。

一句话原则：

```text
小范围、稠密、整数 key：数组。
大范围、稀疏、复杂 key：哈希表。
```

## 排序后降低问题难度

很多问题在原始顺序下很难，但排序后可以利用单调性。

典型场景：

- 合并区间。
- 找重叠关系。
- 双指针求和。
- 贪心选择。

收益：

- 排序成本通常是 `O(n log n)`。
- 排序后可能把原本 `O(n^2)` 的比较降到线性扫描。

取舍：

- 如果必须保留原始顺序，需要记录原下标。
- 排序会改变数据结构，不能用于所有在线场景。

## 用堆维护动态最值

如果需要频繁获取当前最大值或最小值，同时还要插入新元素，可以使用堆。

典型场景：

- Top K。
- 合并 K 个有序序列。
- 任务调度。
- Dijkstra 最短路中的候选节点。

收益：

- 每次取最值 `O(log n)`。
- 比每次重新排序或线性扫描更稳定。

取舍：

- 堆只能高效访问堆顶，不能高效访问任意元素。
- 删除非堆顶元素通常需要额外结构或懒删除。

## 用单调栈或单调队列消除无效候选

如果一个元素在未来永远不可能成为答案，可以尽早丢掉。

单调栈典型场景：

- 下一个更大元素。
- 柱状图最大矩形。
- 去除重复字符并保持字典序。

单调队列典型场景：

- 滑动窗口最大值。
- 固定长度窗口中的最值维护。

收益：

- 每个元素最多入栈或入队一次、出栈或出队一次。
- 很多问题能从 `O(n^2)` 降到 `O(n)`。

取舍：

- 需要证明被弹出的元素以后确实不会更优。
- 维护规则和边界容易写错。

## 先记录位置再处理关系

如果问题中需要频繁查找某个元素的位置，可以先建立反向索引。

典型场景：

- 值到下标的映射。
- 字符到出现位置列表。
- 用户到事件列表。
- 图中节点到邻接边。

收益：

- 把重复扫描变成一次预处理加快速查询。
- 常见复杂度从 `O(nq)` 降到 `O(n + q)` 或 `O(n log n + q log n)`。

取舍：

- 需要额外空间。
- 如果数据频繁变化，索引维护成本会上升。

## 总结清单

遇到复杂度问题时，可以按这个顺序检查：

1. key 范围小吗？小范围优先数组。
2. 状态只是 true/false 吗？考虑 bitset。
3. 有大量区间查询吗？考虑前缀和。
4. 有大量区间更新吗？考虑差分。
5. 连续窗口有单调性吗？考虑双指针。
6. 有重复子问题吗？考虑缓存、记忆化或 DP。
7. 数据稀疏还是稠密？决定哈希表还是数组。
8. 排序后是否出现单调性？考虑排序加扫描。
9. 需要动态最值吗？考虑堆。
10. 有无效候选能提前淘汰吗？考虑单调栈或单调队列。
11. 查询位置很频繁吗？先建反向索引。

这些技巧的本质都是：用更贴近数据特征的表示方式，减少无效搜索、重复计算和不必要的通用结构开销。
